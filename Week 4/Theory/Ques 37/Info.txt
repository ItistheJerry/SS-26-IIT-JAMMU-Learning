Legacy protocols like WPA2 remain highly vulnerable to offline dictionary attacks. An attacker can sniff the initial 4-way cryptographic handshake over the air and use high-compute rigs to brute-force the network password without further network interaction. Furthermore, Evil Twin attacks present a massive threat to automated IoT systems. Because many standard IoT devices simply scan for a matching Service Set Identifier (SSID) and execute a connection, an attacker can spin up a rogue access point with identical network credentials and a stronger signal broadcast. By pairing this setup with a deauthentication flood to disconnect the device from its legitimate hub, the attacker forces the IoT node to hand over its authentication details or reroute its data through a malicious machine, initiating a devastating Man-in-the-Middle (MitM) pipeline. 

Best Practices for Securing ESP32 Connections
When building on hardware like the ESP32 using the ESP-IDF framework, security must be applied at both the network and transport layers.
WPA3-Personal: Upgrading from WPA2 to WPA3 introduces Simultaneous Authentication of Equals (SAE). SAE utilizes a password-authenticated key agreement based on the Diffie-Hellman exchange, making the connection completely immune to offline handshake cracking and mandating Protected Management Frames (PMF) to block unauthorized deauthentication attacks.[1]
Certificate Pinning: At the application layer, standard TLS validation should be augmented with certificate pinning. Instead of relying blindly on a broad chain of public Certificate Authorities (CAs), the ESP32 is flashed with the specific cryptographic public key or certificate hash of its target cloud server. The device explicitly drops the socket connection if the presented certificate deviates by a single bit, successfully neutralising Evil Twin rerouting attempts.[2]
The Danger of Hardcoded Source Credentials
Embedding Wi-Fi SSIDs, passwords, or cloud API tokens directly into compilation variables or raw source code files creates an extreme liability. Standard microcontrollers do not inherently protect their storage medium; if flash encryption is left disabled, any bad actor with transient physical access can attach a basic hardware programmer to the UART or SPI flash pins and extract the production firmware container in seconds. Once extracted, simple reverse-engineering tools instantly output the plaintext parameters. Furthermore, if source files are accidentally tracked into public version control repositories, entire fleets of deployed production hardware can be compromised simultaneously.
